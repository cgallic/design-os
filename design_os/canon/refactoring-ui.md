# Canon: Refactoring UI — Adam Wathan & Steve Schoger

## Why this school is in the canon

*Refactoring UI* (2018) is the most operational design book of the modern web era: written by the creator of Tailwind CSS (Wathan) and its designer (Schoger) explicitly for people who "don't feel like they have design talent," it converts taste into named, numbered moves — spacing scales, type scales, shade counts, shadow recipes, label-removal tests. Schoger's "Little UI Details" tips and his Laracon/CSS Day talks extend the book with concrete before/after fixes. Because nearly every recommendation ships with a number or an enumerable alternative list, this school supplies more directly machine-checkable rules per page than any other source in the canon, and its system-first stance ("limit your choices") maps one-to-one onto design tokens.

## Philosophy

Refactoring UI's central claim is that good design is mostly *decision management*, not talent. An interface looks amateur because its author made hundreds of tiny unconstrained choices — a 13px gap here, a #999 gray there — each individually defensible and collectively incoherent. The fix is to make the choices in advance: define a constrained system of spacing values, font sizes, shades, and shadows, and then "design by process of elimination" — pick the nearest value on the scale, try the neighbors, keep the best. When there are only ten spacing values instead of infinite ones, the difference between candidates is obvious and the result is automatically consistent. Crucially, the scales are perceptual, not mathematical: steps must differ by a meaningful relative amount (small values packed close, large values far apart), and they're defined in absolute units (px/rem), never in relationships (em, percentages) that break the moment context changes.

The second pillar is that *hierarchy is everything*: "visual hierarchy is the most effective tool you have for making something feel 'designed'." But hierarchy is built with the cheap dimensions first — color and weight before size. A secondary detail should get grayer, not tinier; a heading should get heavier, not gigantic. And the counterintuitive core move is subtractive: when the important thing doesn't stand out, don't amplify it — "de-emphasize the elements that are competing with it." Strip the sidebar's background, soften the inactive nav items, mute the icon sitting next to the text. The same subtractive logic governs chrome: labels are "a last resort" (the data's format usually says what it is), borders are a last resort (space, shadows, or background-color shifts separate things more quietly), and not every button deserves a background color (style by position in the action hierarchy, not by semantics).

Finally, design and build are one interleaved loop, done "in cycles": sketch the next feature in low fidelity — grayscale first, so hierarchy is forced to work without color's help — build the simplest real version, discover the actual problems in a working UI, fix, repeat. Don't design states you can't see yet, don't imagine every edge case in the abstract, and "start with too much white space," removing it only where the design earns density. Polish is a distinct final pass — "supercharge the defaults": icon bullets, custom form controls, accent borders, designed empty states — the details that make the same structure read as intentional.

## Workflow practices

1. **Start with a feature, not a layout.** The first thing designed is an actual piece of functionality (the flight-search form, not the app shell); navigation and chrome come after there's something to navigate to.
2. **Work in low fidelity to move fast.** Sketch with a thick Sharpie/marker so detail is impossible; wireframes are disposable exploration tools, not deliverables.
3. **Design in grayscale first.** Force spacing, contrast, and size to carry the hierarchy; add color only once the gray version works.
4. **Design in cycles: design → build → fix → repeat.** Design the simplest version of the next feature, implement it for real, discover the actual problems in a working UI, iterate there, then return to design mode. Don't design every feature or edge case up front — "trying to anticipate every interaction in the abstract leads to frustration."
5. **Ship the happy path first; be a pessimist about scope.** Design the smallest useful version of a feature ("expect it to be hard to build"); leave nice-to-haves for a later cycle rather than blocking on them.
6. **Define the systems before the screens.** Spacing scale, type scale, gray ramp, color ramps, shadow elevations, border radii, opacities — written down as tokens up front, so day-to-day design is "choosing from a small set of options," not inventing values.
7. **Decide by elimination.** To pick a value, take the nearest option on the scale, compare against its neighbors above and below, keep the winner. If two adjacent options are indistinguishable, the scale is too dense.
8. **Start with too much white space,** then remove it until satisfied — never start cramped and try to add air later. (Schoger's input rule of thumb from the same pass: form controls land around 40–48px tall from font size + padding.)
9. **When something looks off, diagnose it as a hierarchy problem** and reach for the subtractive fix first: mute the competitors, remove backgrounds and borders, before amplifying the element you care about.
10. **Run a "supercharge the defaults" polish pass at the end:** replace bullet points with icons, browser-default checkboxes/radios with branded controls, plain blockquotes with pull-quote treatments, add accent borders (a 3–4px colored strip on cards/alerts/active nav) and designed empty states.
11. **Study good interfaces to build the instinct:** when a UI looks great, stop and name exactly what the designer did that you wouldn't have — Schoger's tips are literally this practice published.

## Anti-patterns (reject on sight)

- Arbitrary values: a 13px gap, a 17px font, a 5px radius that appears nowhere else — any spacing/size not on the declared scale.
- Hierarchy done with size alone: giant page titles, 10px gray legalese, and everything between distinguished only by font-size.
- Gray text (`#999`-style, saturation ≈ 0) sitting on a colored button, banner, or hero.
- Font weights under 400 used to de-emphasize text (use a lighter color or smaller size instead).
- `Label: value` litter — "Name: Erin", "In stock: 12" — where the format or a merged phrase would carry the meaning.
- Borders on everything: nested boxes each with 1px outlines, plus a shadow doing the same job; separators where 16px of space would do.
- Every button solid brand-colored; every destructive action a big red primary regardless of context.
- 16px icons scaled to 3–4× ("chunky" hairline-imbalanced icons), and full-page desktop screenshots shrunk to unreadable thumbnails.
- A single ad-hoc `box-shadow` invented per component; shadows with zero or upward offset; glows standing in for elevation.
- One fixed line-height (e.g. 1.5) applied to every size from captions to display headings; full-width paragraphs at 120+ characters per line.
- Centered multi-line body copy; numeric table columns left-aligned in proportional figures.
- Em-based font sizing and percentage-based sidebars — relationships forced to scale together that shouldn't ("don't use percentages unless you actually want it to scale").
- Empty tables rendered as a header row plus "No data", with live-looking filters and sort controls attached.
- Pure black `#000` text and pure untinted gray ramps.
- Designing the whole app in high fidelity before a line of it is built.
